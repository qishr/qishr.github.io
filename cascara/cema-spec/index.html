    This is the final, hardened specification for the Cascara Entity Management Architecture (CEMA). It strips away the fluff, focuses on the mechanical "Press" logic, and sets the stage for a high-integrity, migration-driven ecosystem.

    Cascara Entity Management Architecture (CEMA)

    CEMA is a multi-layered framework for managing data structures and persistence. It is designed to bridge the gap between static, compiled code and dynamic, user-defined entities while ensuring a rigorous, audited database layer.

    1. The Core Hierarchy

    CEMA organizes logic and storage into three distinct tiers:

    * System (Level 1): The immutable foundation. Hard-coded Java classes using io.github.qishr.cascara.lang.schema.annotation (e.g., @SchemaObject, @SchemaField). These define the mandatory traits and core behaviors of the application.

    * Managed (Level 2): The dynamic configuration. Blueprints defined as AST-driven YAML/JSON schemas. These "implement" or extend System traits to create domain-specific entities (e.g., "Tasks," "Bugs") at runtime.

    * Data (Level 3): The persistent payload. Physical SQLite/SQL records governed by the rules of the higher tiers.

    1. Functional Components

    Press (The Orchestrator)

    The central engine responsible for DDL/DML execution. It ensures that the physical Data layer accurately reflects the requirements of the System and Managed tiers.

    Mirror (The State Record)

    A versioned, persisted YAML representation of the physical database schema stored within a system table. It serves as the "Point of Truth" for the current state of the database.

    SchemaMigration (The Interface)

    The formal mechanism for Automated Schema Reconciliation.

    * Role: An interface that defines how to calculate the delta between a blueprint (Level 1/2) and the Mirror.

    * Implementation: Generates and executes the idempotent SQL commands required to synchronize the Data layer.

    Tracked (@Tracked)

    A metadata flag applied to a @SchemaField. It signals to the Press that all state transitions for this field must be captured.

    AttributeLog (The History)

    A physical table in the Data layer storing timestamped entries of every change to a Tracked field.

    * Structure: [EntityID, AttributeID, Value, Timestamp, ActorID]

    Tally (The Metrics)

    A physical table in the Data layer storing daily incremental counts. It provides pre-calculated totals for Tracked values to enable high-speed temporal reporting.

    ValueMap (The Bridge)

    A logical mapping layer that groups specific Managed values into generalized System categories for unified cross-entity reporting.

    1. Operational Workflow

    * Definition: A change occurs in a System class or a Managed schema.

    * Reconciliation: The Press invokes the SchemaMigration interface.

    * Comparison: The current Mirror is compared against the new blueprint.

    * Execution: SQL commands are generated and executed to evolve the Data tables (including AttributeLog and Tally structures if @Tracked fields are present).

    * Persistence: The Mirror is updated to the new version, and the transaction is committed.

    1. Data Integrity Policies

    * Soft Delete: CEMA prohibits physical record deletion. Records are marked with a deleted_at timestamp, ensuring that historical data in the AttributeLog and Tally remains contextually accurate.

    * Visibility Filter: Query-level logic that determines the inclusion of soft-deleted records in active views or reports.

    That makes total sense. To get all three metrics (Created, Deleted, and Total), we need to be slightly more granular in the Tally table.

    If we just use a single count column, "Created" and "Deleted" get squashed together into a "Net Change." To separate them, we should adjust the Tally schema (or at least our logic) to track Increments and Decrements distinctly.

    1. The Schema Adjustment

    Ideally, your Tally table should look like this to support those three metrics:

    • entityType
    • attributeKey
    • attributeValue
    • tallyDate
    • adds (The number created/moved-in on this day)
    • subs (The number deleted/moved-out on this day)

    3. How you get your 3 Metrics

    With this data structure, your SQL queries for a specific day (or range) become very powerful:

    • Number Created: SELECT SUM(adds) FROM Tally WHERE tallyDate = '2026-03-17'
    • Number Deleted: SELECT SUM(subs) FROM Tally WHERE tallyDate = '2026-03-17'
    • Total Outstanding: SELECT SUM(adds) - SUM(subs) FROM Tally (This works across all time)

    4. Integrating with delete

    Now, when you call delete(entity), your processDeletePolicies will call updateTally with a -1. The logic above will see that negative number and increment the subs column for today's date.

    The "Move" Scenario:

    If you change a Bug's severity from LOW to HIGH:

    1. LOW gets a delta = -1 (Incrementing subs for LOW).
    2. HIGH gets a delta = +1 (Incrementing adds for HIGH).
      Your charts will accurately show that one item "left" the LOW category and "entered" the HIGH category today.
