    Schema store notes

    Regarding those "last few ANY labels". Let's make a proper fix that fits in with the extensible nature of Cascara.

    Extension keywords like x-storage are usually defined in their annotation class. For example this one in `io.github.qishr.cascara.persistence.annotation``:

    @Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface Storage {
    Type value() default Type.FOREIGN_KEY;

    public static final String X_STORAGE = "x-storage";

    /// Values for "x-storage" schema extension keyword
    public enum Type {
        FOREIGN_KEY("foreign-key"),
        LINK_TABLE("link-table");
        private final String string;
        Type(String string) { this.string = string; }
        public String string() { return string; }
        public static Type fromString(String s) {
            for (Type strategy : values()) {
                if (strategy.string.equalsIgnoreCase(s)) {
                    return strategy;
                }
            }
            throw new IllegalArgumentException("No enum constant with value " + s);
        }
    }
}

    We have TypeAnalyzers like this one in io.github.qishr.cascara.persistence.core:

    public class PersistenceTypeAnalyzer implements TypeAnalyzer {
    private static final String REF = "$ref";

    @Override
    public void analyze(Field field, SimpleMapNode targetAst) {

        Class<?> type = field.getType();

        // Handle Collections (List, Set) as Link Tables
        if (Iterable.class.isAssignableFrom(type)) {
            targetAst.put(Storage.X_STORAGE, scalar(Storage.Type.LINK_TABLE.string()));
        }
        // Handle standard Reference vs Scalar logic
        else if (targetAst.containsKey(REF) && !isScalarType(type)) {
            targetAst.put(Storage.X_STORAGE, scalar(Storage.Type.FOREIGN_KEY.string()));
        }

        // Handle Annotations

        if (field.isAnnotationPresent(Index.class)) {
            Index ann = field.getAnnotation(Index.class);
            SimpleMapNode indexMeta = new SimpleMapNode();

            // Store the name so the Table Generator can group fields
            indexMeta.put("name", scalar(ann.name()));
            indexMeta.put("unique", scalar(ann.unique()));

            targetAst.put(Index.X_INDEXED, indexMeta);
        }
    }
    ...
}

    The following classes currently have a registerTypeAnalyzer method:

    • io.github.qishr.cascara.schema.util.SchemaBuilder
    • o.github.qishr.cascara.persistence.sqlite.SQLiteSchemaMigrationService
    • io.github.qishr.cascara.schema.util.ClassSchemaGenerator

    Since there is enough information at the instant when a schema is being generated or compiled to be aware of valid extension keywords, we can use this information when building the observable node tree.

    The only question is how we get the information during the `buildTree`` process.

    buildTree is called when a structured document (eg YAML, JSON) is openeed in Cascara Studio. Studio recognizes the document type and creates an AstDataBroker for it. The data broker uses the editor's text buffer, parses the document and builds the tree of observable nodes.

    The only hurdle here is that Studio itelf is unaware of what TypeAnalyzers are used with the schema when the schema is in use. The module using the schema specifies its choice of TypeAnalyzers.

    One possibility is killing three birds with one stone, and creating a schema registry that sits outside the CascaraSchemaResolver and orchestrates the following three things:

    • Which TypeAnalayzers and extension keywords are associated with a schema
    • Making schemas like the CEMA Meta Schema available when there is no running module using CEMA. The schema we've been using with the structural editor (cascara://core/schema-service/cascara.persistence.cema/cema-meta) currently requires the Organizer module (or any other module that uses CEMA) to have been loaded before the Structural Editor (or indeed any editor/consumer) is able to retrieve the meta schema by its $id URI.
    • Associating specific schemas with specific documents. We currently have the following structure in settings.yaml for this, but it could be merged into the "three birds" plan:
      jsonSchemas:
    - files:
        - "/Users/ivor/Spacelift.yaml"
      schema: "/Users/ivor/space-schema.json"

    Getting at least the first two of these points working would be very helpful right now.

    let's not get into any code right now. We need to make sure we have the right approach before we do that. For "bird 2" (Making schemas like the CEMA Meta Schema available when there is no running module using CEMA.) Cascara will need the schemas to be stored on disk since nothing is guaranteed to have access to the module that generated the schema. For "bird 1", for the same reason, the extension keywords (and their types and anything else relevant) will also need to be stored.

    We'll need to formalize the URI syntax for such schemas too.

    As I explained a day or two ago, in Studio, cascara://core/schema-service URIs are handled globally by the ContentLoaderService, which delegates to the CascaraContentService, which routes the request to the appropriate module:

    cascara://core/schema-service/organizer/blah would be routed to the organizer module.

    Perhaps this is the point where we could actually just use the organizer module to supply the necessary extension keyword information, but it would be more efficient and more flexible for it to be stored outside of the module so the module doesn't need to be loaded. Modules should just need to register their extension keyword use when they start up, or something like that.

    No code just now - only discussing what the best approach here is.

    Cascara URIs start off with:

    cascara://<module-name>/

    The module name can be the full JPMS module name, like cascara.module.organizer, or a short form, like organizer and refers to a real JPMS module.

    core is a special module. While it's a module in the JPMS sense, it cannot be unloaded or disabled in Cascara Studio. core provides many interfaces and types that other modules depend on.

    When a cascara://core/ URI is requested, the CascaraContentService looks at the next segment of the URI (technically the first segment of the URI's path) to determine which Cascara Studio Service to delegate the request to. For example, the following URI would be delegates to the SchemaService:

    cascara://core/schema-service/

    The way individual Cascara Studio services handle the rest of the URI has not been formally decided yet.

    The SchemaService currently checks if the next segment of the URI path is a module name, then if the next one is a class name. Then it checks if the class is annotated with @SchemaDefinition, and if it is it uses the `ClassSchemaGenerator`` to create a schema from the class.

    If there is no class specified in the URI, the SchemaService currently delegates to the CascaraSchenaResolver which may have the schema cached already if some user code has called registerSchema.

    At the moment we've been using cascara://core/schema-service/schema/ to indicate schemas that aren't based onb classes and those are likely the ones that should be stored on disk.

    For the manifest/registry design, we need to allow for:

    • Multiuple schemas using the same extension keywords.

    I think keywords should be groupes by owner. Each one can reuse the same keywork. We can't control what 3r party drop-in extensions use as keywords.

    When looking at a schema URI, we should know which extension/module requested that URI to be registered. Therefor we know what keywords that module wants to be used for that schema.

    So the association goes:

    + module/extension
  |
  +- URI
     |
     +- Keywords
     |
     +- Associated files

    The incoming request should maybe be more like:

    cascara://core/schema-service/<module-name>/static/<arbitrary-path>

    Although perhaps static could be replaced with something else. Any ideas?

    That fits in with the existing concept of:

    cascara://core/schema-service/<module-name><class-name> that's currently used for generating schemas from classes with the @SchemaDefinition attribute.

    And is easy to separate from class names at runtime.

    Perhaps we should have: - cascara://core/schema-service/static/<module-name>/<arbitrary-path> - cascara://core/schema-service/dynamic/<module-name>/<class-name>

    WHere dynamic is used for schemas generated from annotated classes, and static is used for schemas supplied by modules at install time or their first run, and could potentially be updated when the module/extension is updated/upgraded.

    resource sounds more like something from a module, which is good. draft would also work if it's intended to support changes. Versioning should probably also be included. draft-<module-version> (Cascara uses SemVer for all modules, so you'd get draft-organizer-1.0.0. This fits in cleanly with both JSON Schema and Maven versioning.

    Then we have the debate of who owns the on-disk schemas. cascara.schema should be able to be used by modules that are not part of Cascara Studio. All of the other modules like cascara.lang.xml and cascara.persistence.sqlite, even cascara.ui can be used independently.

    This means the ownership should be in cascara.schema even if it is not CascaraSchemaResolver itself. This makes it look like we need a class in cascara.schema that handles on-disk schemas. We could even put cascara.schema's cached JSON Meta Schemas in there, removing an extra level of complexity from the resolver.

    So for these 3 URIs:

    • cascara://core/schema-service/dynamic/organizer/cema-meta (generated at run-time)
    • cascara://core/schema-service/draft/organizer/1.0.0/cema-meta (registered at install of first run)

    And draft schemas must have a version.
